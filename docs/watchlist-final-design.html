C 03 "내 관심 랏" 동작

저장 (전체 교체 · set semantics)

  • 화면의 lot_id 리스트 = 내 watchlist 전체. 저장 시 내 행 전부 삭제 후 재삽입(UnitOfWork 원자성)
  • row 지우고 저장 → watchlist에서 빠짐 (멱등)
  • 빈 row drop · 중복 lot_id dedupe · order_index로 입력 순서 보존

검증 · 표시 (저장 허용 → 표시 시 해석)

  • lot_id 무조건 저장(유저 의도 보존). 표시 시 lots live JOIN
  • 매칭 ✓ → status/step 등 채움 · 매칭 ✕ → "조회 대기/없음" 행 유지
  • 다음 dump에서 자동 채워짐 (JOIN이 매번 최신 lots 참조)

갱신 트리거 (합본)

① 저장 직후 재조회 ② 탭 포커스 복귀 재조회 (백그라운드 throttle 보완) ③ lot_dump_meta.last_run_at 가벼운 폴링 → 변경 시에만 watchlist 재조회 (dump 위상 정렬)
  • blind 30분 타이머 ✕ (위상 어긋남·백그라운드 throttle 회피) → heartbeat 추종으로 dump 완료 직후 자동 반영
  • 03 신선도 = "데이터 기준: HH:MM" 라벨 (= last_run_at), dump 죽으면 ⚠️. (02 같은 신호등 아님)

D 테이블 & DDL (pholex가 docs+seed에 정의 → 사내 AI가 적용)

-- 30분 dump 마스터 캐시 (사내 AI 소유 · pholex는 읽기만). 02 fallback + 03 공용 소스
CREATE TABLE lots (
  lot_id        VARCHAR PRIMARY KEY,
  status        VARCHAR,           -- run | wait | hold
  equipment     VARCHAR,
  process_step  VARCHAR,
  hold_comment  TEXT,
  hold_operator VARCHAR,           -- ⚠ CONTRACT-1: 실제 컬럼명 다름(예 lot_hold_user_id), 값=사번
  updated_at    TIMESTAMPTZ
);

-- pholex 고유 도메인 (dump와 무관). 유저↔lot 매칭
CREATE TABLE user_lot_watchlist (
  employee_number VARCHAR,         -- 사번 (users FK)
  lot_id          VARCHAR,
  order_index     INT,             -- 입력 순서 보존
  PRIMARY KEY (employee_number, lot_id)
);

-- dump 생존 heartbeat. 1행 고정. 신선도 판정 단일 소스
CREATE TABLE lot_dump_meta (
  id          INT PRIMARY KEY DEFAULT 1,
  last_run_at TIMESTAMPTZ NOT NULL,    -- ⚠ CONTRACT-4: 사내 dump가 매 실행 now로 upsert
  row_count   INT,
  status      VARCHAR DEFAULT 'ok'
);
