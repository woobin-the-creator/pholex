LotSource vs LotRepository — 어느 포트에 lot_status 읽기를 둘까

실시간 API가 사내에도 미구현 → 30분 dump(lot_status)가 슬롯1 "내 hold"의 primary로 확정.
그렇다면 dump 테이블을 읽는 코드를 어느 포트(콘센트)에 꽂을 것인가.
🚰

LotSource / 원천

데이터를 가져온다 — 물을 길어오는 취수구
방향바깥(사내) → 앱
메서드fetch_my_holds() 내 hold 가져오기
subscribe_changes() 변경 스트림
성격"어디서 데이터를 얻나"
VS
🛢️

LotRepository / 보관소

데이터를 보관했다 꺼낸다 — 받아둔 물의 물탱크
방향앱 ↔ 앱의 DB
메서드upsert_lots_batch() 저장
get_my_holds_cached() 꺼내기
성격"그걸 어디에 쟁여두나"
질문 — lot_status의 "내 hold" 읽기를 어디에 꽂나?
lot_status는 성격이 둘 다다: 사내가 채웠으니 취수구 같고, pholex DB 테이블이니 탱크 같다.
(A) LotSource 포트에
RealLotSource.fetch_my_holds가 lot_status를 읽게 한다.
→ usecase·wire 무변경. 포트 의미만 "실시간"→"dump 읽기"로 해석.
(B) LotRepository 포트에
읽기 메서드를 새로 추가하고 usecase가 source 대신 그걸 호출.
→ usecase 분기 변경 + 포트 2개 손댐.
추천 (A). ① 헥사고날의 핵심 = "같은 포트에 다른 어댑터 갈아끼우기" → 나중에 실시간 API가 생겨도 윗단 무변경 교체. ② fetch_my_holds usecase·wire를 안 건드려 변경 최소. ③ reference 구현 PgSampleLotSource가 이미 이 패턴(테이블→LotRowDTO)이라 사내가 베끼기 쉬움.
주의 — (A)로 가도 신호등(health)은 이 포트로 안 들어온다. fetch_my_holds는 hold 목록만 반환하고 last_run_at(탱크 받은 시각)은 안 싣는다 → 신선도는 별도 결정(다음 질문).
