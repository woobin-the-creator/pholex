## 1. 증상과 결론 (먼저 읽기)

**증상**: `ADAPTER_MODE=real`에서 "내 lot hold" 조회 시 500 / 504 에러.

**오해 정정**: "전체 테이블을 가져와서 과부하"가 아닙니다. `fetch_my_holds`의 쿼리는 이미 `WHERE lot_status_seg = 'Hold'` + `LIMIT 100`이 있고, `sample` 테이블은 14,316행짜리 로컬 Postgres 테이블이라 쿼리는 10ms 미만입니다. **쿼리는 범인이 아닙니다.** 그러니 페이지네이션을 추가하지 마세요 — 증상이 안 사라집니다.

**진짜 원인 3가지 (전부 실제 버그이므로 전부 고칩니다)**:

| ID | 원인 | 결과 | 수정 위치 |
|----|------|------|-----------|
| **S1** | `cache_my_holds`가 `json.dumps`에 datetime 객체를 넘김 → `TypeError` | **500** | `lot_repository.py` |
| **S2** | `create_async_engine`를 요청마다 새로 생성, dispose 안 함 → 커넥션 풀 고갈 | **504** (요약 4.8의 "QueuePool limit reached") | `lot_repository.py`, `lot_source.py`, `container.py` |
| **S3** | Redis 연결 에러를 try/except 안 함 → cache 호출이 요청을 죽임 | **500** | `lot_repository.py` |

→ **Phase 1~3을 전부 적용**하면 500/504가 해결됩니다. 어느 게 먼저 터지는지 몰라도 됩니다 (셋 다 버그라 다 고칠 가치 있음).

추가로 **S4 (correctness, 500 원인 아님)**: "내 lot hold"가 employee 필터 없이 전체 hold를 반환 중 → Phase 4에서 다룸 (별도, 정보 필요).

## Phase 1 — S1 수정: 캐시 datetime 직렬화 (500 해결, 최우선)

**파일**: `backend/app/adapters/real/lot_repository.py`

### 1-1. `cache_my_holds` — `model_dump()` → `model_dump(mode="json")`

**원인**: `LotRowDTO.model_dump()`는 `updated_at`을 `datetime` **객체**로 반환합니다. `json.dumps`는 datetime을 직렬화할 수 없어 `TypeError`를 던지고, 이게 UnitOfWork 안에서 터지면 요청이 500으로 죽습니다. `mode="json"`을 주면 datetime이 ISO 8601 **문자열**로 변환됩니다.

**BEFORE**:
```python
async def cache_my_holds(self, employee_number: str, rows: list[LotRowDTO]) -> None:
    """사번별 hold 결과 캐시 적재. 후속 get_my_holds_cached 에서 hit."""
    if not self._redis_client:
        return

    cache_key = f"pholex:my_holds:{employee_number}"
    serialized = json.dumps([row.model_dump() for row in rows])
    await self._redis_client.set(cache_key, serialized)
```

**AFTER**:
```python
async def cache_my_holds(self, employee_number: str, rows: list[LotRowDTO]) -> None:
    """사번별 hold 결과 캐시 적재. 후속 get_my_holds_cached 에서 hit."""
    if not self._redis_client:
        return
    try:
        cache_key = f"pholex:my_holds:{employee_number}"
        # mode="json" 필수: datetime → ISO 문자열로 변환해야 json.dumps 가 동작한다.
        serialized = json.dumps([row.model_dump(mode="json") for row in rows])
        # TTL 60초: live 대시보드라 오래된 캐시를 들고 있으면 안 된다.
        await self._redis_client.set(cache_key, serialized, ex=60)
    except Exception as e:
        # 캐시는 best-effort. 실패해도 요청을 죽이지 않는다 (다음 호출은 그냥 cache miss).
        logger.warning("cache_my_holds skipped (cache error): %s", e)
```

### 1-2. `get_my_holds_cached` — redis 에러를 cache miss로 처리

**BEFORE**:
```python
async def get_my_holds_cached(self, employee_number: str) -> list[LotRowDTO] | None:
    """캐시된 결과 반환. None 은 *cache miss* (key 부재). 빈 리스트는 *정상 빈 결과*."""
    if not self._redis_client:
        return None

    cache_key = f"pholex:my_holds:{employee_number}"
    cached_json = await self._redis_client.get(cache_key)

    if cached_json is None:
        return None

    cached_data = json.loads(cached_json)
    return [LotRowDTO.model_validate(item) for item in cached_data]
```

**AFTER**:
```python
async def get_my_holds_cached(self, employee_number: str) -> list[LotRowDTO] | None:
    """캐시된 결과 반환. None 은 *cache miss* (key 부재 또는 캐시 장애). 빈 리스트는 *정상 빈 결과*."""
    if not self._redis_client:
        return None
    try:
        cache_key = f"pholex:my_holds:{employee_number}"
        cached_json = await self._redis_client.get(cache_key)
        if cached_json is None:
            return None
        cached_data = json.loads(cached_json)
        # model_validate 는 ISO 문자열 updated_at 을 datetime 으로 다시 파싱한다 (1-1과 짝).
        return [LotRowDTO.model_validate(item) for item in cached_data]
    except Exception as e:
        # 캐시 장애는 miss 로 간주 → fetch 경로로 폴백. 절대 요청을 죽이지 않는다.
        logger.warning("get_my_holds_cached treated as miss (cache error): %s", e)
        return None
```

> ⚠️ `logger`가 이 파일에 정의돼 있어야 합니다. 파일 상단에 없으면 추가:
> ```python
> import logging
> logger = logging.getLogger(__name__)
> ```

## Phase 2 — S2 수정: 엔진 싱글톤화 (504 해결)

**문제**: `RealLotRepository.__init__`이 `create_async_engine()`를 호출하는데, DI가 요청마다 새 `RealLotRepository`를 만들면 **요청마다 새 커넥션 풀(size 5 + overflow 10)** 이 생기고 **절대 닫히지 않습니다.** 요청이 쌓이면 DB 커넥션이 고갈되어 504. (요약 4.8에서 직접 관측된 증상.) `RealLotSource`도 자체 엔진을 만들면 풀이 2배가 됩니다.

**해결 원칙**: **엔진은 프로세스당 1개만.** 모듈 레벨 싱글톤으로 만들고, 모든 repository/source 인스턴스가 그 하나를 공유합니다. repository 객체 자체는 요청마다 새로 만들어도 됩니다 (엔진을 안 만드니 가벼움).

### 2-1. 공유 엔진 모듈 생성 (신규 파일)

**신규 파일**: `backend/app/adapters/real/_engine.py`

```python
from __future__ import annotations

from functools import lru_cache

from sqlalchemy.ext.asyncio import (
    AsyncEngine,
    AsyncSession,
    async_sessionmaker,
    create_async_engine,
)

from app.config import settings  # settings.DATABASE_URL 이 있는 위치. 실제 import 경로에 맞게 수정.


@lru_cache(maxsize=1)
def get_engine() -> AsyncEngine:
    """프로세스당 단 하나의 async 엔진. 절대 요청마다 만들지 말 것."""
    return create_async_engine(
        settings.DATABASE_URL,
        echo=False,
        pool_size=10,        # 상시 커넥션
        max_overflow=20,     # 피크 시 추가 허용
        pool_pre_ping=True,  # 죽은 커넥션 자동 감지/교체 (장시간 운영 필수)
        pool_recycle=1800,   # 30분마다 커넥션 재활용 (방화벽/DB idle timeout 회피)
    )


@lru_cache(maxsize=1)
def get_session_factory() -> async_sessionmaker[AsyncSession]:
    """공유 엔진에 바인딩된 세션 팩토리. 모든 repository/source 가 이걸 쓴다."""
    return async_sessionmaker(
        get_engine(),
        class_=AsyncSession,
        expire_on_commit=False,
    )
```

> ⚠️ `from app.config import settings` 의 실제 경로는 사내 코드 기준으로 맞추세요. `settings.DATABASE_URL`을 어디서 읽는지 기존 `lot_repository.py` 상단을 보고 동일하게.

### 2-2. `RealLotRepository.__init__` — 자체 엔진 생성 제거

**파일**: `backend/app/adapters/real/lot_repository.py`

**BEFORE**:
```python
def __init__(self, database_url: str | None = None, redis_url: str | None = None) -> None:
    self._database_url = database_url or settings.DATABASE_URL
    self._redis_url = redis_url or settings.REDIS_URL

    self._engine = create_async_engine(self._database_url, echo=False)
    self._session_factory = async_sessionmaker(self._engine, class_=AsyncSession, expire_on_commit=False)

    self._session: AsyncSession | None = None

    self._redis_client = None
    if self._redis_url:
        import redis.asyncio as redis
        self._redis_client = redis.from_url(self._redis_url, decode_responses=True)
```

**AFTER**:
```python
def __init__(self, database_url: str | None = None, redis_url: str | None = None) -> None:
    self._redis_url = redis_url or settings.REDIS_URL

    # 엔진을 여기서 만들지 않는다. 공유 싱글톤 팩토리를 사용한다 (S2 수정).
    from app.adapters.real._engine import get_session_factory
    self._session_factory = get_session_factory()

    self._session: AsyncSession | None = None

    self._redis_client = None
    if self._redis_url:
        import redis.asyncio as redis
        self._redis_client = redis.from_url(self._redis_url, decode_responses=True)
```

> 변경점: `self._engine = create_async_engine(...)` **삭제**. `self._session_factory`를 공유 팩토리로 대체. `__aenter__`/`__aexit__`/`commit`/`rollback`은 그대로 둡니다 (세션을 닫는 건 맞고, 닫으면 안 되는 건 엔진).
> 파일 상단의 `from sqlalchemy.ext.asyncio import create_async_engine` import가 더 이상 안 쓰이면 제거해도 됩니다 (안 해도 무방).

### 2-3. `RealLotSource`도 동일하게 공유 팩토리 사용

**파일**: `backend/app/adapters/real/lot_source.py`

`fetch_my_holds`가 `async with self._session_factory() as session:`을 쓰고 있습니다. 이 `self._session_factory`가 **자체 엔진**에서 만들어졌다면 그것도 S2 누수원입니다. `__init__`에서 자체 엔진 생성을 제거하고 공유 팩토리로:

**`RealLotSource.__init__` 안에서 (현재 코드를 확인 후)**:

```python
# BEFORE (자체 엔진을 만들고 있었다면):
# self._engine = create_async_engine(...)
# self._session_factory = async_sessionmaker(self._engine, ...)

from app.adapters.real._engine import get_session_factory
self._session_factory = get_session_factory()
```

> ⚠️ `RealLotSource.__init__`의 실제 코드가 요약본에 없습니다. **현재 `__init__` 전체를 사용자에게 보고**하고, 자체 `create_async_engine`가 있으면 위처럼 공유 팩토리로 교체하세요. `sample` 테이블이 `lots`와 **같은 Postgres(DATABASE_URL)** 면 같은 엔진을 공유하면 됩니다. **다른 DB**면 `_engine.py`에 두 번째 싱글톤 함수를 추가하세요.

## Phase 3 — S3 마무리: Redis 존재 여부 정리

Phase 1에서 redis 호출을 try/except로 감쌌으니 이제 redis가 죽어도 500은 안 납니다. 하지만 **캐시가 아예 동작 안 하면** 매 요청이 fetch+upsert를 타서 부하가 늘 수 있습니다. 둘 중 하나를 고르세요:

### 옵션 3-A — redis 컨테이너 추가 (권장, 캐시 살림)

**파일**: `docker-compose.yml` (또는 사내가 쓰는 compose 파일)
```yaml
  redis:
    image: ${DOCKER_REGISTRY:-}redis:7-alpine
    ports:
      - "10008:6379"   # 호스트 포트는 사내 포트 정책에 맞게
```
그리고 `backend` 서비스의 `depends_on`에 `redis` 추가:
```yaml
  backend:
    depends_on:
      - postgres
      - redis
```
`.env.dev`의 `REDIS_URL=redis://redis:6379/0`은 그대로 두면 됩니다 (컴포즈 서비스명 `redis`로 접속).

### 옵션 3-B — redis 비활성 (캐시 끄고 단순화)

`.env.dev`에서 `REDIS_URL`을 **빈 값**으로:
```bash
REDIS_URL=
```
그러면 `self._redis_client = None`이 되어 캐시 경로를 전부 건너뜁니다. 매 요청이 DB를 타지만, hold 100건 쿼리는 가벼우니 MVP에선 문제없습니다. **빠르게 안정화하려면 이쪽.**

## Phase 4 — S4 (correctness, 500 원인 아님): employee 필터

**현재 문제**: `fetch_my_holds`가 `employee_number`를 **무시**하고 전체 hold 중 100건을 반환합니다. 그래서 "내 lot hold"가 **모든 사람의 hold**를 보여줍니다. 사용자가 "전체를 가져온다"고 느낀 정체가 이것입니다.

**막힌 점**: 요약에 "sample 테이블에는 employee_number 컬럼이 없음"이라고 돼 있습니다. 그럼 **무엇으로 hold 담당자를 식별하는지** 알아야 필터를 짤 수 있습니다.

**그래서 사용자에게 질문 (답이 와야 진행)**:
1. `sample` 테이블 90개 컬럼 중 **hold 담당자/소유자/요청자**를 나타내는 컬럼이 있나요? (예: `hold_user_id`, `operator`, `charge_emp`, `req_user` 등) 컬럼명만 알려주세요.
2. 없다면, **사번 → 담당 lot** 매핑이 어디에 있나요? (별도 테이블? 외부 시스템?)
3. 그것도 없다면 MVP에서 "내 hold"의 정의를 어떻게 할까요? (전체 표시 유지 / 특정 라인만 / 추후 구현)

**1번에 컬럼이 있다고 가정한 패치 템플릿** (컬럼명 `<OWNER_COL>` 자리에 실제 컬럼명 대입):
```python
query = text("""
    SELECT
        lot_id, lot_status_seg, eqp_type, step_name,
        lot_hold_comment, last_update_date
    FROM sample
    WHERE lot_status_seg = 'Hold'
      AND <OWNER_COL> = :emp        -- ← employee 필터 추가
    ORDER BY lot_id ASC
    LIMIT 100
""")
result = await session.execute(query, {"emp": employee_number})
```
그리고 컬럼/인덱스: `<OWNER_COL>`에 인덱스가 없으면 사내 DBA와 협의해 `(lot_status_seg, <OWNER_COL>)` 복합 인덱스 추가 권장 (14k행이라 급하진 않음).

> **답을 모르면 Phase 4를 건너뛰고 Phase 1~3만 적용해서 먼저 500/504를 잡으세요.** Phase 4는 별도로 진행해도 됩니다.

## Phase 5 — 위생 (여유 되면)

이번 장애와 직접 관련은 없지만 같이 정리하면 좋은 것들:

1. **중복 alembic 마이그레이션 제거**: `versions/001_create_lots_table.py`와 `versions/initial_create_lots_table.py`가 **둘 다 lots 테이블을 생성**합니다. 마이그레이션 체인이 깨질 수 있으니 **하나만 남기세요** (down_revision 체인 확인). 어느 걸 남길지 모르면 `alembic history`를 사용자에게 보고.

2. **`upsert_lots_batch` 빈 리스트 가드**: `rows`가 비어 있으면 `pg_insert(LotTable).values([])`가 깨질 수 있습니다. 함수 시작에 추가:
   ```python
   if not rows:
       return
   ```

3. **`fetch_my_holds`의 광범위 except 정리**: `except Exception: return []`이 **모든 DB 에러를 "hold 없음"으로 위장**해서 진단을 어렵게 만듭니다. 최소한 에러 로그는 이미 남기지만, 가능하면 연결 에러(재시도 대상)와 데이터 에러(스킵 대상)를 구분하세요. (이번엔 안 해도 됨, 메모만.)

4. **`_map_status_to_canonical` 실제값 반영**: 현재 매핑은 **추측한 enum**("RUNNING"/"HOLDING"...)입니다. 실제 분포를 확인:
   ```sql
   SELECT DISTINCT lot_status_seg, COUNT(*) FROM sample GROUP BY lot_status_seg;
   ```
   결과를 사용자에게 보고하고 매핑 테이블을 실제값으로 교체. (주석에 섞인 중국어 `根据实际情况`도 제거.)

```bash
# 1) 재빌드 + 기동 (의존성/compose 바뀌었으면 --build)

for i in $(seq 1 50); do

    "http://localhost:10005/api/lots/my-hold?force_refresh=true"
done; echo
#   기대: 200 이 50번. 504/500 섞이면 S2 미해결 → container.py / lot_source.py 엔진 재확인

# 4) 백엔드 로그에 TypeError / QueuePool 없는지
docker compose -p pholex-dev logs backend 2>&1 | grep -i "typeerror\|queuepool\|traceback" | tail -20
#   기대: 빈 출력

# 5) 프론트 확인
#   http://localhost:10004 → "내 lot hold" 패널에 데이터 표시, 새로고침 반복해도 안 죽음

# 6) CI gate — 보호 디렉터리 안 건드렸는지 (반드시 빈 diff)
git diff origin/main -- backend/app/domain backend/app/usecases backend/app/api backend/app/ports
#   기대: 빈 출력
```
